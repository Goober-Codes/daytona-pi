## Problem Statement

Daytona has two architectural gaps that limit what AI agents can safely do inside its sandboxes:

**Gap 1 — Network Security:** Daytona's network policy is 440 lines of Go wrapping iptables (`apps/runner/pkg/netrules/`). CIDR-based allowlists only. No HTTP-level inspection, no secret mediation, no per-request hooks. A sandbox with access to `api.github.com` can read env vars and POST them anywhere — Daytona has no way to prevent it architecturally.

**Gap 2 — Workflow Durability:** Daytona's runner is a poll-execute loop (`apps/runner/pkg/runner/v2/poller/` + `executor/`). No durable multi-step workflows, no crash recovery for agent tasks, no event-driven suspension. If a multi-step agent task (generate → test → human review → deploy) crashes at step 3, everything reruns from scratch.

Two open source projects by Armin Ronacher fill these gaps exactly: **Gondolin** (programmable VM-level network proxy) and **Absurd** (durable workflows on Postgres).

## What Makes This Cool

**The composability nobody else has built:** Gondolin's JS `onRequest` hook and Absurd's `awaitEvent()` are the same abstraction — a synchronous checkpoint in an otherwise async execution. Wire them together and you get:

- Every outbound HTTPS call from a sandbox emits an Absurd event
- The workflow suspends, waiting for a policy decision
- A rule/human/LLM resolves it
- The request proceeds, gets secret-replaced, or gets dropped
- The entire decision chain is checkpointed in Postgres — a full forensic audit trail

oshu-vault does egress control as a hosted service. DBOS does durable workflows. Nobody has checkpointed network policy decisions as first-class durable workflow events.

## Premises

1. **Gondolin wraps Docker (confirmed from source code):** `docker.json` + `docker-init-extra.sh` show that Gondolin runs a QEMU micro-VM with Docker daemon inside. Docker containers inside the VM have their HTTPS intercepted by Gondolin's network stack at the VM boundary. The MITM CA bundle is auto-injected via a wrapped `docker run` command.

2. **Absurd shares Daytona's Postgres:** Absurd is a single `absurd.sql` schema file. Drop it into Daytona's existing Postgres instance. No new services. Schema verification needed before touching production DB.

3. **oshu-vault is competition (as a hosted service):** oshu-vault already does hosted secret injection + egress control for Daytona/E2B. Differentiation: Gondolin is open source, embedded at the VM level, not a hosted sidecar. This is architecturally stronger (can't be bypassed by raw sockets inside the container).

4. **Learning-first, sequential:** Absurd first (TypeScript, familiar stack), Gondolin second (requires Go runner changes). Months, not a weekend.

## Approaches Considered

### Approach A: Standalone PoC — Gondolin + Absurd, no Daytona codebase
- Docker Compose: Gondolin (Node.js) + Absurd worker (TypeScript) + Postgres
- Gondolin provisions a QEMU VM with Docker inside
- Absurd workflow: provision VM → run agent → `awaitEvent("network.policy.decision")` → resume
- Gondolin `onRequest` hook emits Absurd event
- Kill worker mid-run → restart → resumes from last step
- Effort: S | Risk: Low | Completeness: 7/10

### Approach B: Absurd into Daytona API only
- Apply `absurd.sql` to Daytona's Postgres
- Replace `job.service.ts` Redis queue with Absurd `spawn_task`/`claim_task`
- Map Daytona webhook events → `awaitEvent()`
- Purely TypeScript/NestJS, no Go changes
- Effort: M | Risk: Medium | Completeness: 8/10

**The core pattern (what makes this worth publishing):**

Important architectural note: Gondolin's `onRequest` hook is a real async callback — it CAN `await` a Promise. But Absurd's `ctx.awaitEvent()` works by throwing a `SuspendTask` exception, which exits the entire task handler. You cannot call `ctx.awaitEvent()` inside the `onRequest` hook without crashing the VM host. Instead: the hook polls Postgres directly for a policy decision, written there by the `policy-resolver.ts` CLI. The Absurd workflow checkpoints the final VM execution result. The `network_events` table is the audit trail — separate from Absurd's step checkpointing, but in the same Postgres DB.

```typescript
// src/absurd-worker.ts
absurd.registerTask({ name: "sandboxed-agent" }, async (params, ctx) => {
  const result = await ctx.step("run-in-vm", async () => {
    const vm = await VM.create({
      httpHooks: createHttpHooks({
        allowedHosts: params.allowedHosts,
        secrets: params.secrets,
        onRequest: async (req) => {
          // Write pending decision to shared Postgres table
          await pg.query(
            `INSERT INTO network_events (request_id, url, method, status)
             VALUES ($1, $2, $3, 'pending')`,
            [req.id, req.url, req.method],
          );
          // Real async poll — not a SuspendTask throw
          const decision = await pollForDecision(req.id, pg);
          if (!decision.allow) throw new Error(`blocked: ${req.url}`);
          return req;
        },
      }),
    });
    const output = await vm.exec(params.command);
    await vm.close();
    return output;
  });
  return result;
});

// Poll Postgres until policy-resolver.ts writes a decision
async function pollForDecision(requestId: string, pg: Pool) {
  while (true) {
    const { rows } = await pg.query(
      `SELECT status FROM network_events WHERE request_id = $1`,
      [requestId],
    );
    if (rows[0]?.status === "approved") return { allow: true };
    if (rows[0]?.status === "denied") return { allow: false };
    await sleep(100);
  }
}
```

```typescript
// src/policy-resolver.ts — run in a second terminal
// Reads from network_events (status='pending'), prompts stdin, writes decision
const { rows } = await pg.query(
  `SELECT request_id, url, method FROM network_events WHERE status = 'pending' LIMIT 1`,
);
if (rows[0]) {
  const allow = await promptUser(`Allow ${rows[0].method} ${rows[0].url}? [y/n] `);
  await pg.query(
    `UPDATE network_events SET status = $1 WHERE request_id = $2`,
    [allow ? "approved" : "denied", rows[0].request_id],
  );
}
```

**Known limitation:** If the Absurd worker crashes mid-`run-in-vm` step, the QEMU process is orphaned (no `VM.attach()` static method exists to reconnect across process restarts). The step replays from scratch on resume, provisioning a fresh VM. For the PoC this is fine — document it in the README.

**README one-liner for Armin:**
> "Gondolin's `onRequest` JS hook and Absurd's `awaitEvent()` are the same abstraction — a synchronous checkpoint in an otherwise async execution. This repo wires them together."

**What makes this worth starring:** run `docker compose up`, then in a second terminal run `node policy-resolver.ts` to approve/deny live network requests from a sandboxed agent. Kill the worker — restart — it resumes from the last checkpoint. The full decision trail is in Postgres.

**Public artifact:** A fork at `github.com/[you]/daytona` with a branch `feat/absurd-workflow-engine` and an open draft PR against `daytonaio/daytona`.

**Read first:**
```
apps/api/src/sandbox/services/job.service.ts         — Redis BRPOP queue (what you're replacing)

apps/api/src/webhook/services/                       — where lifecycle events are emitted
apps/runner/pkg/runner/v2/poller/                    — Go runner polls the HTTP API, NOT Redis
```

**Important:** The Go runner never touches Redis. It polls the Daytona API over HTTP (`PollJobs`). Only `job.service.ts` in the NestJS API uses Redis BRPOP. **The Go runner does not change in Phase 2.**

**What changes in job.service.ts** — three things, not one:
1. Remove `@InjectRedis()` constructor parameter + `InjectRedis` import
2. Remove `BullModule`/`RedisModule` from the NestJS module imports
3. Add `AbsurdModule` + `Absurd` instance — replace `BRPOP` dispatch with `absurd.spawn()` / `absurd.startWorker()`

Webhook service (`apps/api/src/webhook/`) calls `absurdApp.emitEvent()` on `sandbox.created` and `sandbox.state.updated`.

**Apply `absurd.sql` as a raw TypeORM migration** using the `query()` runner — not auto-generated entity DDL. It's 1,400 lines of PL/pgSQL. Wrap it as:
```typescript
await queryRunner.query(fs.readFileSync('migrations/absurd.sql', 'utf8'));
```

**Durable sandbox lifecycle:**
```typescript
absurd.registerTask({ name: "sandbox-lifecycle" }, async (params, ctx) => {
  await ctx.step("provision", () => sandboxService.create(params.id));
  await ctx.awaitEvent(`sandbox.started.${params.id}`);   // emitted by webhook service
  await ctx.step("execute", () => runAgentCode(params));
  await ctx.awaitEvent(`sandbox.stopped.${params.id}`);
  await ctx.step("destroy", () => sandboxService.destroy(params.id));
});
```

**The PR description should say:** "Replaces Redis BRPOP job queue with Absurd durable workflows on the existing Postgres instance. Sandboxes that crash mid-lifecycle resume from the last completed step. No new infrastructure — just `absurd.sql` applied to the existing DB. Go runner is unchanged. Latency tradeoff: Redis BRPOP dispatch is ~1ms; Absurd's pull model adds ~100ms configurable poll interval."

```
Daytona Runner
  └─ GondolinClient.Create(sandboxDto)
       └─ VM.create({ httpHooks, secrets, allowedHosts })
            └─ QEMU micro-VM
                 ├─ Docker daemon (running inside VM)
                 ├─ Agent code (Docker containers)
                 └─ Gondolin network stack (HTTPS intercepted at VM boundary)
                      └─ onRequest → Absurd emitEvent → workflow suspends
```

The iptables approach can be bypassed by raw sockets inside the container. VM-level interception cannot. This is the architectural argument for the PR.

## Open Questions

1. **Postgres schema conflicts:** Does Daytona's existing schema have any table name conflicts with `absurd.`-prefixed tables? Needs verification before Phase 2.
2. **QEMU on runner hosts:** Daytona runs on Linux (Debian/Ubuntu). Does the runner host expose `/dev/kvm`? QEMU without KVM acceleration will be significantly slower than Docker containers.
3. **Absurd pull vs. Redis BRPOP:** Daytona's API uses blocking Redis BRPOP for low-latency job dispatch (~1ms). Absurd's polling model adds ~100ms configurable poll interval. Whether this latency is acceptable for sandbox provisioning UX needs input from the Daytona team before the Phase 2 PR.
4. ~~**Gondolin licensing:** Apache-2.0. Daytona is AGPL-3.0.~~ Apache-2.0 is permissive and compatible with AGPL-3.0. Not a blocker.
